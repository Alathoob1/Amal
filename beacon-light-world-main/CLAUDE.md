# CLAUDE.md — Design System Rules (Figma MCP Integration)

Project: **Hemaya** — bilingual (AR/EN) autism-support platform.
Stack: **TanStack Start** (React 19, file-based routes) + **Vite 8** + **Tailwind CSS v4** + **shadcn/ui** ("new-york" style) + **Radix UI** primitives, built/managed via **Lovable**.

This doc tells an agent (or a Figma → code pipeline) exactly where things live and what patterns to follow so generated code matches the existing system instead of inventing a new one.

---

## 1. Design Tokens

**Source of truth:** `src/styles.css` — a single Tailwind v4 CSS file. There is no separate JSON/YAML token file and no Style Dictionary–style transform pipeline; tokens are authored directly as CSS custom properties and consumed via Tailwind's `@theme inline` block.

### Structure

- **Raw values** are defined as CSS variables on `:root` (light) and `.dark` (dark mode), using the **OKLCH** color space exclusively (no hex/rgb in tokens):
  ```css
  :root {
    --radius: 1rem;
    --background: oklch(0.985 0.010 85);
    --primary: oklch(0.52 0.055 155);      /* sage — primary */
    --secondary: oklch(0.93 0.025 85);     /* sand — secondary */
    --accent: oklch(0.72 0.085 45);        /* terracotta — used sparingly */
    --warm: oklch(0.82 0.075 75);          /* dusty gold — badges */
    --success: oklch(0.62 0.075 150);
    --info: oklch(0.62 0.055 220);
    --destructive: oklch(0.58 0.130 25);
    ...
  }
  .dark { /* same variable names, dark-mode values */ }
  ```
- **Semantic aliasing** happens in `@theme inline { ... }`, which maps each raw variable to a Tailwind utility color (`--color-primary: var(--primary)`, etc.), radii (`--radius-sm` … `--radius-4xl`, all derived from the single `--radius` value), fonts, and two custom shadow tokens (`--shadow-soft`, `--shadow-lifted`).
- **Fonts** are tokens too, with separate LTR/RTL pairs:
  - `--font-sans: "Manrope"`, `--font-display: "Fraunces"` (LTR)
  - `--font-sans-ar: "IBM Plex Sans Arabic"`, `--font-display-ar: "Readex Pro"` (RTL, swapped in automatically — see §7)

### Rules for adding/editing tokens
- Never hardcode a color, radius, or shadow value in a component. Always reference the Tailwind utility that maps to the token (`bg-primary`, `text-muted-foreground`, `rounded-2xl`, `shadow-soft`), not raw OKLCH/hex.
- New tokens must be added in **three places**: `:root`, `.dark`, and the `@theme inline` alias block — all three must stay in sync.
- Palette is intentionally **muted/desaturated** (low chroma values, ~0.01–0.13) — this is a deliberate sensory-friendly constraint for an autism-focused product. Do not introduce high-saturation colors when translating new Figma frames; flag it if a Figma design uses vivid/saturated color and reconcile with the existing token instead of importing a new raw value.
- `--radius` is a single base value; all other radii cascade from it. Changing roundedness project-wide means changing one variable.

## 2. Component Library

**Location:** `src/components/`, split into two tiers:

- `src/components/ui/` — shadcn/ui primitives (Button, Card, Dialog, Select, Sidebar, Tabs, etc.), one file per component, unmodified-by-default shadcn output. This is the "atoms" layer generated by the shadcn CLI.
- `src/components/site/` — hand-written, app-specific composite components (`site-nav.tsx`, `site-footer.tsx`, `page-header.tsx`, `language-switcher.tsx`). This is where Figma-derived, page-level/composite components should go.

### Architecture pattern (shadcn/ui primitives)
- `React.forwardRef` components, typed via `React.HTMLAttributes<...>` or a CVA-derived props interface.
- Variants defined with **`class-variance-authority` (CVA)**, not styled-components/emotion:
  ```tsx
  const buttonVariants = cva(
    "inline-flex items-center justify-center gap-2 rounded-md text-sm font-medium ...",
    {
      variants: {
        variant: { default: "...", destructive: "...", outline: "...", secondary: "...", ghost: "...", link: "..." },
        size: { default: "h-9 px-4 py-2", sm: "...", lg: "...", icon: "h-9 w-9" },
      },
      defaultVariants: { variant: "default", size: "default" },
    },
  );
  ```
- Class composition always goes through the `cn()` helper (`src/lib/utils.ts` — `clsx` + `tailwind-merge`), never string concatenation, to safely resolve conflicting Tailwind classes.
- `asChild` (via `@radix-ui/react-slot`) is the standard pattern for polymorphic components (e.g. rendering a `Button` as a `Link`).
- Radix UI is the underlying primitive layer for anything with accessibility complexity (dialogs, dropdowns, tooltips, accordions) — do not hand-roll these; extend the existing Radix-based component.

### Composite/site components pattern
- Function components (not `forwardRef`) that pull translated strings via `useLanguage()`/`useT()` (see §6) and compose `ui/` primitives + Tailwind utility classes directly. `page-header.tsx` is a good reference for the expected shape of a new page-level component (typed props, optional `eyebrow`/`description`, `children` slot).
- No Storybook, no `.mdx` docs, no visual test/snapshot setup in this repo. `components.json` (shadcn config) is the closest thing to a component manifest — check it before assuming a shadcn component doesn't exist yet:
  ```json
  { "style": "new-york", "iconLibrary": "lucide", "aliases": { "components": "@/components", "ui": "@/components/ui", "hooks": "@/hooks", "lib": "@/lib" } }
  ```

### Rules
- When a Figma component maps to an existing shadcn primitive (button, card, badge, dialog, etc.), reuse it and only add variants via CVA — do not create a parallel one-off component.
- New page-specific composites belong in `components/site/`, imported via the `@/` alias, following the `page-header.tsx` prop-typing style.

## 3. Frameworks & Libraries

| Concern | Choice |
|---|---|
| UI framework | React 19 (`react`, `react-dom` ^19.2.0) |
| Routing | TanStack Router / TanStack Start (file-based routes in `src/routes/`, generated `src/routeTree.gen.ts`) |
| Styling | Tailwind CSS v4 (`@tailwindcss/vite` plugin — CSS-first config, no `tailwind.config.js`) |
| Component system | shadcn/ui ("new-york" style) on top of Radix UI primitives |
| Variant/class management | `class-variance-authority` + `clsx` + `tailwind-merge` |
| Icons | `lucide-react` |
| Forms | `react-hook-form` + `@hookform/resolvers` + `zod` |
| Animation utility | `tw-animate-css` |
| Data/async | `@tanstack/react-query` |
| Build tool | Vite 8, via `@lovable.dev/vite-tanstack-config` (wraps Tailwind, TanStack Start, path aliasing, Nitro server build — **do not manually re-add these plugins**, see `vite.config.ts` header comment) |
| Package manager | Bun (`bun.lock`, `bunfig.toml`) |
| Server target | Nitro, defaulting to Cloudflare |

`vite.config.ts` is intentionally minimal:
```ts
import { defineConfig } from "@lovable.dev/vite-tanstack-config";
export default defineConfig({
  tanstackStart: { server: { entry: "server" } },
});
```
Almost all bundler concerns are centralized in the shared Lovable config — extend via the `vite`/other keys of `defineConfig`, don't fork or duplicate plugin setup.

There is **no CSS-in-JS library** (no styled-components/emotion) — styling is Tailwind utility classes exclusively.

## 4. Asset Management

- Static images live in `src/assets/` (currently just `hero-autism.jpg`) and are **imported as ES modules**, not referenced by string path:
  ```tsx
  import heroImage from "@/assets/hero-autism.jpg";
  // ...
  <img src={heroImage} alt={t.home.heroImgAlt} />
  ```
  This lets Vite hash/optimize the file at build time. Follow this pattern for any new image asset introduced from a Figma export — do not put content images in `public/` and reference by raw path.
- `public/` is reserved for files that must be served verbatim at a fixed URL: `favicon.ico`, `robots.txt`, and the generated `sitemap.xml` route.
- No image CDN, no `next/image`-style optimization component, and no explicit responsive `srcset` pipeline exists yet — images are plain `<img>` tags with Tailwind sizing classes. If Figma exports need multiple resolutions, that pipeline would need to be introduced (currently absent).
- Alt text is **always sourced from the i18n dictionary** (`t.home.heroImgAlt` etc.), never hardcoded — see §6.

## 5. Icon System

- **Library:** `lucide-react` exclusively (also declared as the icon library in `components.json`: `"iconLibrary": "lucide"`). No custom SVG icon set, no icon sprite sheet, no `react-icons`.
- **Import pattern:** named imports of individual icons, tree-shaken:
  ```tsx
  import { Menu, X, Heart } from "lucide-react";
  import { Languages } from "lucide-react";
  ```
- **Usage pattern:** rendered directly as JSX components with Tailwind sizing/opacity utilities, `aria-hidden` when purely decorative next to text:
  ```tsx
  <Heart className="h-4 w-4" aria-hidden />
  <Languages className="h-3.5 w-3.5 opacity-70" aria-hidden />
  ```
- Default sizing baked into `buttonVariants`: any `<svg>` inside a `Button` auto-sizes to `size-4` via `[&_svg]:size-4 [&_svg]:shrink-0` — don't manually resize icons inside buttons unless intentionally overriding.
- **Naming convention:** use Lucide's own PascalCase export names as-is (`Menu`, `X`, `Heart`, `Languages`) — no local aliasing/renaming layer.
- When translating a Figma icon to code, map it to the closest existing Lucide icon before considering a custom SVG. Only add a bespoke SVG if there is truly no Lucide equivalent, and if so keep it as an inline `<svg>` using `currentColor`/theme tokens so it respects dark mode and RTL automatically.

## 6. Styling Approach

- **Methodology:** utility-first Tailwind CSS v4, CSS-first configuration (no `tailwind.config.ts`/`.js` — theme, tokens, and custom utilities are all authored directly in `src/styles.css`).
- **Global styles:** `src/styles.css`, structured as:
  1. `@import "tailwindcss" source(none)` + `@source` scoping + `tw-animate-css` import
  2. `@theme inline { ... }` — token → Tailwind utility mapping (see §1)
  3. `:root` / `.dark` — raw token values
  4. `@layer base { ... }` — element resets: border color default, body font/background/color, heading font-family override (`font-display`), `::selection` styling, and a **`prefers-reduced-motion` global kill-switch** that collapses all animation/transition durations to ~0 — this is a deliberate accessibility/sensory requirement, do not remove it.
  5. Custom `@utility` blocks — reusable composed utilities, e.g.:
     ```css
     @utility container-page {
       margin-inline: auto; width: 100%; max-width: 76rem; padding-inline: 1.25rem;
       @media (min-width: 768px) { padding-inline: 2rem; }
     }
     @utility surface-card {
       background: var(--color-card); border: 1px solid var(--color-border);
       border-radius: var(--radius-2xl); box-shadow: var(--shadow-soft);
     }
     ```
     Prefer using/extending these two utilities (`container-page` for page-width wrappers, `surface-card` for elevated content blocks) over re-deriving the same combination of classes inline.
  6. RTL typography overrides (see §7).
- **Responsive design:** standard Tailwind breakpoint prefixes (`md:`, `lg:`) directly in JSX class strings, mobile-first (base classes = mobile, prefixed classes override upward). No container queries or custom breakpoint tokens beyond Tailwind defaults. A dedicated `useIsMobile()` hook (`src/hooks/use-mobile.tsx`, 768px threshold) exists for JS-level responsive branching when a CSS-only solution isn't enough — prefer CSS/Tailwind breakpoints first, reach for this hook only when conditional rendering (not just styling) is required.
- **No CSS Modules, no scoped `.module.css` files, no styled-components** — everything is Tailwind utility classes composed via `cn()`.

## 7. Internationalization & RTL (critical — treat as part of the design system, not an afterthought)

Every visual pattern in this repo is expected to work in **both** Arabic (RTL) and English (LTR). This is not optional polish.

- **Dictionary:** `src/lib/i18n-dict.ts` — a single typed object (`Lang = "ar" | "en"`, `dictionaries.en` / `dictionaries.ar`) exporting a `Dict` type inferred from the `en` object's shape, namespaced by area (`nav`, `footer`, `home`, `page`, etc.).
- **Provider/hook:** `src/lib/i18n.tsx` — `LanguageProvider` (persists choice to `localStorage` under `hemaya.lang`, defaults to `"ar"`, sets `lang`/`dir` attributes on `<html>`) and `useLanguage()` / `useT()` hooks.
- **Consumption pattern in components:**
  ```tsx
  const { t } = useLanguage();
  <Link to="/about">{t.nav.about}</Link>
  ```
- **Rule: every user-facing string must come from `t.*`.** No literal English/Arabic text in JSX. The one enforced exception is the literal brand wordmark "Hemaya" itself. When porting a new Figma frame to code, every label/placeholder/aria-label/button text must first be added to **both** `en` and `ar` blocks in `i18n-dict.ts`, then referenced via `t`.
- **RTL layout is driven by CSS logical properties + the `dir` attribute, not per-direction utility variants.** The codebase already uses `gap-*`, `items-*`, `justify-*`, and flexbox default document order rather than `ml-*/mr-*/left-*/right-*`, so layouts mirror correctly under `dir="rtl"` automatically. **Continue this convention**: never introduce `ml-`, `mr-`, `pl-`, `pr-`, `left-`, `right-` for anything that should flip in RTL — use `gap-*`, `space-x-*` is also physical and should be avoided in favor of `gap-*` on flex/grid, or Tailwind's logical variants (`ps-*`, `pe-*`, `ms-*`, `me-*`) if a one-sided offset is unavoidable.
- **Typography swaps automatically for RTL** via attribute selectors in `styles.css`, not per-component logic:
  ```css
  html[dir="rtl"] { font-family: var(--font-sans-ar); letter-spacing: 0; }
  html[dir="rtl"] h1, h2, h3, h4, .font-display { font-family: var(--font-display-ar); font-weight: 600; }
  ```
  Do not hardcode font-family in a component — this global rule already handles it.

## 8. Project Structure

```
src/
  routes/            # File-based routes (TanStack Router). One file per page:
                      #   index, about, ai-analysis, community, contact, dashboard,
                      #   faq, knowledge, login, places, register, __root.tsx (shell),
                      #   sitemap[.]xml.ts (generated route)
  components/
    ui/               # shadcn/ui primitives — generic, reusable, no app-specific copy
    site/             # App-specific composite components (nav, footer, header, language switcher)
  lib/
    i18n.tsx          # LanguageProvider + hooks
    i18n-dict.ts      # AR/EN string dictionary (source of truth for all copy)
    utils.ts          # cn() helper
    error-capture.ts, error-page.ts, lovable-error-reporting.ts  # Lovable SSR error tooling
  hooks/
    use-mobile.tsx    # viewport breakpoint hook
  assets/             # imported image assets (ES module imports, not public/ paths)
  styles.css          # single global stylesheet: tokens, base layer, custom utilities, RTL rules
  router.tsx, server.ts, start.ts, routeTree.gen.ts   # TanStack Start wiring (generated/framework files)
public/               # verbatim-served static files (favicon, robots.txt)
components.json       # shadcn/ui config (style, aliases, icon library)
vite.config.ts        # thin wrapper around @lovable.dev/vite-tanstack-config
```

### Feature organization pattern
- Organization is **by architectural layer**, not by feature/domain: all primitives in `components/ui`, all routes flat in `routes/`, all copy centralized in one dictionary file. There are no per-feature folders (e.g. no `features/dashboard/` grouping). When adding a new page from Figma:
  1. Add a route file in `src/routes/` (TanStack Router file-based convention).
  2. Add any new composite component to `components/site/` (or `components/ui/` only if it's a truly generic, content-agnostic primitive).
  3. Add all new copy to both language blocks in `src/lib/i18n-dict.ts`.
  4. Reuse `PageHeader`/`container-page`/`surface-card` for structural consistency with existing pages rather than inventing new page-shell markup.
- Path alias `@/*` → `./src/*` (`tsconfig.json`), matching shadcn's `aliases` in `components.json` (`@/components`, `@/lib`, `@/ui`, `@/hooks`).

## Quick checklist for turning a Figma frame into code here

1. Colors/spacing/radii → map to existing tokens (`bg-*`, `text-*`, `rounded-*` utilities from `styles.css`); don't invent new raw values unless the token genuinely doesn't exist, and if so add it to `:root` + `.dark` + `@theme inline`.
2. Any interactive element with an existing shadcn/Radix equivalent (button, dialog, dropdown, tabs, tooltip, etc.) → reuse `components/ui/*`, extend via CVA variant, don't rebuild.
3. Icons → find the closest `lucide-react` icon first.
4. Images → `import` into `src/assets/`, not a raw path.
5. Every string → added to `src/lib/i18n-dict.ts` in both `en` and `ar`, consumed via `useLanguage()`/`useT()`. Zero literal copy in JSX.
6. Layout → logical/flex-gap based, no physical `left/right/ml/mr` utilities, so it mirrors correctly in `dir="rtl"`.
7. Motion → respect the global `prefers-reduced-motion` kill-switch; keep transitions calm/subtle by default (this is a sensory-friendly product).
8. New page → file in `src/routes/`, built from `PageHeader` + `container-page`/`surface-card` utilities for structural consistency.
